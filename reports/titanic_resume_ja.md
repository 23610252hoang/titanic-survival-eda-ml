# レジュメ：Titanic生存予測

## 1. 目的

本課題では，Titanicの乗客データを用いて，乗客が生存したかどうかを予測するモデルを作成した。単にAccuracyを高めることだけではなく，元データを確認し，EDAによって仮説を立て，特徴量エンジニアリングを行い，検証用データで結果を評価することを目的とした。

## 2. 元データの確認

使用したデータは以下の通りである。

- `train_local (1).csv`：学習用データ，623行，12列，`Survived`あり
- `valid (1).csv`：検証用データ，134行，12列，`Survived`あり
- `eval (1).csv`：提出用データ，134行，11列，`Survived`なし
- `sample_submission.csv`：提出形式のサンプル

主な列は，`PassengerId`，`Survived`，`Pclass`，`Name`，`Sex`，`Age`，`SibSp`，`Parch`，`Ticket`，`Fare`，`Cabin`，`Embarked`である。

`train_local`では，以下の列に欠損値があった。

- `Age`：119件
- `Cabin`：487件
- `Embarked`：2件

## 3. EDAで注目した点

まず，目的変数`Survived`の分布を確認した。学習データでは，生存者は約38.4%，非生存者は約61.6%であった。

次に，`Survived`と各列の関係を確認した。

- `Sex`：女性の生存率は約75.1%，男性の生存率は約18.7%であり，性別は非常に重要な変数であると考えた。
- `Pclass`：1等客室の生存率は約63.9%，2等客室は約45.4%，3等客室は約24.9%であり，客室等級によって生存率が異なっていた。
- `Embarked`：乗船港によって生存率に差が見られた。
- `Cabin`：欠損が多いが，客室情報があるかどうか自体が乗客の属性を表す可能性があると考えた。
- `Name`：氏名に含まれる敬称は，性別，年齢，社会的立場を表す可能性があると考えた。

## 4. 欠損値処理

欠損値処理は`Pipeline`の中で行った。

- 数値列（`Age`，`Fare`など）は中央値で補完した。
- カテゴリ列（`Sex`，`Embarked`，`Title`，`Deck`など）は最頻値で補完した。
- `Cabin`は欠損が非常に多いため，客室番号をそのまま使うのではなく，`HasCabin`と`Deck`に変換して利用した。

中央値を使った理由は，平均値より外れ値の影響を受けにくいためである。カテゴリ列については，最も一般的な値で補完する最頻値を用いた。

## 5. カテゴリ変数の変換

文字列やカテゴリ変数は，そのままではモデルに入力できないため，One-Hot Encodingによって数値化した。

対象とした主なカテゴリ変数は以下である。

- `Sex`
- `Embarked`
- `Title`
- `Deck`

また，検証用データや提出用データに学習時には存在しなかったカテゴリが出現してもエラーにならないように，`handle_unknown="ignore"`を指定した。

## 6. 新しく作成した特徴量

EDAで得られた仮説にもとづき，以下の特徴量を作成した。

- `FamilySize`：`SibSp + Parch + 1`で作成した。本人を含めた家族人数を表す。
- `IsAlone`：`FamilySize == 1`の場合に1とした。一人で乗船していたかどうかを表す。
- `Title`：`Name`から敬称を抽出した。性別，年齢，社会的立場を表す可能性がある。
- `HasCabin`：`Cabin`が欠損していない場合に1とした。客室情報の有無を表す。
- `Deck`：`Cabin`の先頭文字を取り出した。船内の位置に関係する可能性がある。

これらの特徴量は，家族構成，客室情報，社会的属性が生存率に関係している可能性があるという仮説にもとづいて作成した。

## 7. ベースラインとの比較

最初のベースラインモデルでは，以下の列を使用した。

- `Pclass`
- `Sex`
- `Age`
- `Fare`
- `Embarked`

検証用データ`valid.csv`に対するAccuracyは以下の通りである。

| 実験名 | Valid Accuracy |
| --- | ---: |
| baseline_logreg | 0.805970 |
| baseline_rf | 0.820896 |
| feature_gb | 0.850746 |
| feature_logreg | 0.873134 |
| feature_rf | 0.873134 |

特徴量を追加した結果，`feature_logreg`および`feature_rf`でAccuracyが`0.873134`となり，ベースラインより改善した。

## 8. うまくいった点

- `Sex`，`Pclass`，`Title`など，生存率と関係がありそうな変数をEDAで確認できた。
- `FamilySize`，`IsAlone`，`Title`，`HasCabin`など，解釈しやすい特徴量を作成できた。
- `train_local`，`valid`，`eval`に同じ前処理を適用できるように関数化した。
- `Pipeline`を用いることで，欠損値処理とカテゴリ変数変換を一貫して行うことができた。

## 9. うまくいかなかった点・改善点

- ハイパーパラメータの調整は十分には行っていない。
- Cross Validationによる安定性の確認はまだ行っていない。
- `AgeGroup`や`FareGroup`などの区間化特徴量は，今後さらに試す余地がある。
- `Ticket`からグループ情報を抽出するなど，追加の特徴量エンジニアリングも可能である。

## 10. 今後の改善案

今後は，Cross Validationを導入し，モデルの汎化性能をより安定して評価したい。また，Random ForestやGradient Boostingのパラメータ調整，`AgeGroup`，`FareGroup`，`TicketGroup`などの特徴量追加，特徴量重要度の確認を行うことで，モデルの改善と説明性の向上を目指したい。

## 11. 提出ファイル

`eval.csv`に対する予測結果として，`submission_eval.csv`を作成した。このファイルは以下の2列を持つ。

- `PassengerId`
- `Survived`

行数は`eval.csv`と同じ134行であり，`Survived`は0または1のみである。
